## PROMPT 1 — Build the scan plugin

```
## Context (carry forward)

Project: HUDCustomizer — a MelonLoader mod for MENACE (Unity 6, Il2Cpp, .NET 6).
Template to follow exactly: CombatFlyoverPlugin from FloatingCombatText.cs.
Plugin interface: IModpackPlugin with OnInitialize(MelonLogger.Instance, HarmonyLib.Harmony),
OnSceneLoaded(int, string), OnUpdate().

Hard constraints that MUST be followed:
- All game types use Il2CppMenace.* prefix at compile time. Dummy DLL Menace.* names are reference only — never use them.
- harmony.PatchAll(typeof(YourPatchClass)) always. Never bare harmony.PatchAll().
- Every Harmony patch body MUST be wrapped in try/catch. Never crash the game.
- Do NOT use foreach on Il2Cpp lists. Use indexed for loops.
- Do NOT call TacticalEventHooks.Initialize or Intercept.Initialize — both fail silently on this build.
- Do NOT access game singletons in OnInitialize — scene is not loaded yet.

Prior finding being confirmed: Ghidra decompilation of GameAssembly.dll showed LineOfSightVisualizer pools
child GameObjects containing standard Unity LineRenderer components, not a Shapes library. The earlier
runtime scan was wrong because of a bad component resolution path, not missing bindings. This plugin
confirms four open questions from LineOfSightVisualizer-AI.md:
  Q1. Does GetComponentsInChildren<LineRenderer>(true) reach the pooled renderers at all?
  Q2. What are the exact Il2Cpp type names and GameObject names of every component on each child?
  Q3. How many LineRenderer components exist per child? (Resize() loops 3 iterations — unknown if that is
      3 renderers per child, 3 children with 1 renderer each, or something else.)
  Q4. What are the default startColor and endColor values on the renderers?

## Task

Create a new standalone scan plugin LOSScanPlugin.cs.
Mirror the FloatingCombatText.cs project structure exactly. Do NOT modify any existing file.

The plugin MUST contain:

### LOSScanPlugin.cs

One class implementing IModpackPlugin. In OnInitialize, apply exactly one Harmony patch class:
a postfix on Il2CppMenace.Tactical.LineOfSightVisualizer.SetVisible(bool).

The postfix MUST:
1. Guard with `if (_scanned) return;` at the very top.
2. Run the following scan steps in order, logging every result with the prefix [LOSScan]:
   a. Call Object.FindObjectsOfType<Il2CppMenace.Tactical.LineOfSightVisualizer>() and log the count.
   b. Take the first instance. Log instance.transform.childCount.
   c. For i = 0 to childCount-1: log child name (instance.transform.GetChild(i).gameObject.name) and
      the count of components on it via GetComponents<Il2CppInterop.Runtime.InteropTypes.Il2CppObjectBase>()
      or Component — use whichever compiles. Log each component's GetIl2CppType().FullName.
   d. Call instance.GetComponentsInChildren<UnityEngine.LineRenderer>(true). Log the count returned.
   e. For each LineRenderer returned: log its gameObject.name, transform.parent?.gameObject.name,
      startColor.ToString(), and endColor.ToString().
   f. Call instance.GetComponentsInChildren<UnityEngine.Component>(true). Log every entry's
      GetIl2CppType().FullName and gameObject.name. This catches renderers under any type alias.
3. Set _scanned = true after all logging completes.
4. Wrap the entire body in try/catch — log exceptions as MelonLogger Warning.

In OnSceneLoaded, reset _scanned = false (re-fire on next tactical scene load).
OnUpdate is empty.

All log lines MUST use the prefix [LOSScan] for grep-ability.

### LOSScanPlugin.cs

Copy FloatingCombatText.cs exactly. Change AssemblyName and RootNamespace to LOSScanPlugin.
Keep all assembly references identical — do not add or remove any.

## Deliverables — exactly this one files, nothing else:
- LOSScanPlugin.cs

## Stop and ask before:
- Adding any reference not already in FloatingCombatText.cs.
- Creating any file outside LOSScanPlugin.cs.
- Modifying FloatingCombatText.cs or any other existing file.
```
